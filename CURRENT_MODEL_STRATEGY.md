# 当前冰壶 AI 模型与策略说明

## 1. 文档目的

本文描述当前比赛机器人从服务器状态到最终出手的完整决策链，包括状态表示、先后手双模型、战术库、连续搜索、擦冰、官方物理校准、训练方式、运行参数、证据、局限和后续路线。

本文以当前代码为准。训练研究仓库为 `DCCourse`，服务器最小运行仓库为 `curling-runtime`。

## 2. 当前系统结论

当前机器人不是单一端到端强化学习模型，而是一个分层决策系统：

```text
官方服务器状态
  -> 先手/后手身份判断
  -> 88 维状态向量
  -> 对应侧的策略价值网络
  -> 合法战术候选排序
  -> 战术库生成基础投掷参数
  -> 本地连续参数搜索
  -> BESTSHOT(v0, h0, w0)
  -> MOTIONINFO
  -> 可选 SWEEP(distance)
```

| 层 | 职责 | 当前实现 |
| --- | --- | --- |
| 协议层 | 连接服务器、解析状态、发送动作 | `AIRobot.py` |
| 状态层 | 把石位和比赛信息编码为模型输入 | `tactic_dqn_robot.py` |
| 战术层 | 从 16 种战术中排序候选 | 先手/后手 `PolicyValueNet` |
| 参数层 | 将战术转换为 `(v0,h0,w0)` | `tacticslib/` |
| 搜索层 | 在基础参数附近搜索并估计局面得分 | `continuous_shot_search.py` |
| 物理层 | 快速预测落点、简化碰撞和局末得分 | `fast_curling_env.py` |
| 擦冰层 | 搜索或启发式决定擦冰距离 | `curling_sweep.py` |

核心设计原则是：**网络负责缩小战术空间，搜索负责决定具体出手。** 网络不直接输出连续投掷参数。

## 3. 博弈与状态建模

### 3.1 Markov 状态

每个局面视为一个 Markov 状态。有效状态应包含足够信息，使后续决策不再依赖更早历史。当前状态包含石位、投壶序号、先后手身份、当前得分质量等信息。

如果扩展到正式多局比赛，还应更明确地加入累计比分、剩余局数和 hammer 转移；当前模型对这些跨局信息表达较弱。

### 3.2 88 维状态向量

`make_state_vector()` 输出 `STATE_DIM=88`：

1. 前 80 维：最多 16 个冰壶，每个冰壶 5 个特征；
2. 后 8 维：当前投壶和局面摘要。

每个冰壶的 5 个特征：

```text
x 坐标
y 坐标
到大本营圆心的距离
先手/后手标识
当前是否为有效得分壶
```

场上冰壶按距圆心从近到远排序；不存在或出界的冰壶以零填充。服务器返回 `(0,0)` 时，当前实现将其视为未投掷或已出界。

附加 8 维：

```text
shot_num / 15
本方第几壶 / 8
是否先手
next_shot
本方视角当前得分 / 8
本方最近壶质量
对方最近壶质量
场上壶数量和局数的压缩特征
```

所有得分均转换为当前玩家视角，避免后手模型把先手得分方向学反。

## 4. 战术动作空间

当前策略网络在 16 个离散战术中选择：

| 序号 | 战术 | 含义 |
| ---: | --- | --- |
| 0 | `draw_center` | 中路进营 |
| 1 | `guard_left` | 左侧保护/占位 |
| 2 | `guard_right` | 右侧保护/占位 |
| 3 | `curl_left` | 左旋进营 |
| 4 | `curl_right` | 右旋进营 |
| 5 | `occupy` | 战术库占位 |
| 6 | `take_out` | 击飞目标壶 |
| 7 | `hit_roll` | 击打后甩入目标区域 |
| 8 | `defense` | 防守性落位 |
| 9 | `freeze` | 粘贴目标壶 |
| 10 | `clear` | 清空/清障 |
| 11 | `middle_in_center` | 中路进营 |
| 12 | `push_in` | 传进/推进 |
| 13 | `push_in_14` | 特定后段推进 |
| 14 | `double_hit_gote` | 双飞/双击后手壶 |
| 15 | `defense_push_in` | 防守推进组合 |

前五种包含固定基础动作，其余动作调用课程战术库。战术库可能因当前石位或规则条件返回不可行动作；运行时建立合法动作掩码，网络只能在合法战术中选择。

若全部战术均不可用，机器人回退到：

```text
BESTSHOT 3.0 0.0 0.0
```

## 5. 先手与后手独立模型

当前默认模型：

```text
model/search_distill_tactic_policy_first.pt
model/search_distill_tactic_policy_second.pt
```

兼容回退模型：

```text
model/search_distill_tactic_policy.pt
```

服务器通过 `CONNECTNAME Player1/Player2` 告知机器人身份：

- Player1/先手使用 first 模型；
- Player2/后手使用 second 模型。

先后手不能混成一个无身份策略，因为：

- 后手拥有最后一壶，终局价值与先手不同；
- 相同石位在两个身份下的进攻/防守价值不同；
- 后手通常追求两分或空局，先手通常限制对方得分或偷分；
- 最后一壶的 hammer 搜索预算只属于后手。

本地相同配置评测中，独立 second 模型把后手平均分从 `3.5625` 提升到 `3.7375`；双侧平均分从 `3.2500` 提升到 `3.3375`。该证据来自旧本地模拟器，只能证明身份分离方向合理，不能代表官方比赛强度。

## 6. 策略价值网络

### 6.1 网络结构

`PolicyValueNet` 是 CPU 可运行的双头全连接网络：

```text
88 维输入
  -> Linear(88,256) + ReLU
  -> Linear(256,256) + ReLU
  -> Policy head: Linear(256,16)
  -> Value head: Linear(256,1) + tanh
```

Policy head 输出 16 个战术 logit。Value head 输出 `[-1,1]` 的局面价值。

当前比赛推理主要使用 Policy head 做候选排序；Value head 会被前向计算，但没有直接进入最终连续搜索打分。最终搜索仍使用 rollout 结束后的局末得分。

### 6.2 网络不是连续控制器

网络不会直接输出速度、偏移、角速度或擦冰距离。它回答的是：

> 在当前状态下，哪些战术最值得进一步计算？

具体参数来自战术库，并由连续搜索在局部邻域中调整。

## 7. 训练方法

### 7.1 Search Distillation

训练采用搜索蒸馏：

1. 在本地快速环境中生成一局；
2. 轮到受控玩家时枚举全部合法战术；
3. 对每个战术做若干 rollout；
4. 用最终局末得分估计战术价值；
5. 将搜索价值 softmax 化为 policy target；
6. 将搜索最优战术作为 action target；
7. 将本局最终得分除以 8 作为 value target；
8. 用这些目标训练策略价值网络。

搜索目标温度当前为 `0.55`。

### 7.2 损失函数

```text
loss = policy_distillation_loss
     + 0.35 * best_action_cross_entropy
     + 0.50 * value_mse
```

优化器为 AdamW，权重衰减 `1e-4`，梯度范数裁剪为 `4.0`。

### 7.3 对手来源

| 模式 | 对手 |
| --- | --- |
| `random` | 原本地简单随机对手 |
| `scripted` | 按比分和阶段选择手写优先级 |
| `rollout` | 低成本分数感知策略 |
| `model` | 对手侧已有模型 |
| `model-mix` | model/scripted/rollout/random 的偏强混合 |
| `balanced-mix` | random 50%、model 20%、scripted 15%、rollout 15% |

训练游戏和搜索 teacher 使用同一个 opponent context，避免实际训练对手变强但 teacher 仍按随机对手评估的错位。

### 7.4 当前正式模型状态

当前正式 first/second 模型来自较大规模的旧训练：

- first 模型本质上是原 first-player 搜索蒸馏模型；
- second 模型使用 `--player second` 独立训练；
- 当前正式模型主要基于原本地随机对手物理分布。

后来训练过 60 局、2 rollout、8 epoch 的 `model-mix` 候选：

- first 候选与旧 first 基本持平，提升不足以证明；
- second 候选平均分 `2.10`，低于旧 second 的 `2.40`；
- 因此候选没有替换正式模型。

`balanced-mix` 已实现并 smoke 验证，但尚未完成足够规模、且基于官方校准物理的正式重训，所以没有上线。

## 8. 比赛时决策流程

### 8.1 状态接收

机器人按协议处理：

```text
CONNECTKEY
CONNECTNAME
ISREADY
NEWGAME
SETSTATE
POSITION
GO
MOTIONINFO
SCORE
GAMEOVER
```

收到 `GO` 后计算并发送 `BESTSHOT`；收到 `MOTIONINFO` 后决定是否发送 `SWEEP`。

### 8.2 模型候选排序

运行时先执行全部 16 个战术函数，得到当前合法动作。然后：

1. 选择当前身份对应模型；
2. 对 88 维状态做前向推理；
3. 屏蔽非法动作；
4. 按 logit 排序，取 top-k；
5. 保留一个 scripted fallback 候选，防止模型候选过于单一。

代码中存在按先后手、局面阶段和当前比分计算的策略先验，但 `--strategy-gate-strength` 默认是 `0.0`。此前实验未证明该先验能稳定提升，因此当前正式策略不启用手工 gate。

### 8.3 连续参数搜索

对每个候选战术的基础参数，在邻域中生成 `v0/h0/w0/sweep` 扰动。

典型低速候选邻域：

```text
dv: -0.16, -0.08, 0, +0.08, +0.16
dh: -0.09, -0.045, 0, +0.045, +0.09
dw: 旋转壶使用 -0.45, 0, +0.45
sweep: 0、基础值、基础值±2m
```

高速击打、禁止擦冰战术和部分推进战术使用更保守的参数邻域。

候选超过预算时：

- 70% 预算保留距基础动作最近的核心候选；
- 剩余预算从较远候选中随机探索；
- 基础动作不会因循环顺序被意外丢弃。

### 8.4 Rollout 评价

每个候选动作在 `FastCurlingEnv` 中执行，然后把剩余 16 壶模拟完。候选得分为当前玩家视角的局末比分。

最终按风险调整分数选择：

```text
candidate_score = mean_end_score - 0.04 * std_end_score
```

低 rollout 数时标准差信息有限。例如 `rollouts=1` 时 `std=0`，只适合连接和时间预算 smoke，不等同于高质量搜索。

### 8.5 自适应搜索预算

| 阶段 | top-k | candidates | rollouts |
| --- | ---: | ---: | ---: |
| normal | 3 | 24 | 2 |
| late | 4 | 32 | 3 |
| hammer | 至少 4 | 48 | 4 |

`late` 用于本方最后两壶；`hammer` 只用于后手最后一壶。`--fixed-search` 会关闭阶段预算。

服务器首次 smoke 常用：

```text
--fixed-search --search-candidates 16 --search-rollouts 1
```

该配置优先验证协议和耗时，不应作为最终强度结论。

## 9. 官方物理校准

### 9.1 原 mock 问题

原本地无碰撞落点模型：

```text
x = 2.375 + 0.88*h + 0.55*tanh(w/5)
y = 8 - 1.02*v + 0.05*abs(w)
```

在 190 条官方无碰撞、无擦冰、在场内样本上，总 RMSE 约 `1.09m`，主要误差来自 y。错误物理会使连续搜索越算越偏。

### 9.2 当前落点拟合

当前校准拟合：

```text
(v0,h0,w0) -> (final_x,final_y)
```

特征：

```text
1, v, h, w, |w|, tanh(w), v², h², w², v*h, v*w, h*w
```

x 和 y 分别最小二乘拟合。200 条样本中：

- 190 条在场内样本用于拟合；
- 10 条出界样本不参与落点回归；
- 验证集总 RMSE 约 `0.038m`；
- x 残差标准差约 `0.0166m`；
- y 残差标准差约 `0.0323m`。

配置：`config/physics_calibration.json`。拟合脚本：`fit_physics_calibration.py`。

### 9.3 校准有效范围

```text
v0: 2.753 .. 3.236
h0: -0.450 .. 0.432
w0: -1.485 .. 1.467
```

运行时允许 `0.15` support margin，超出范围回退旧 mock，不做危险外推。因此：

- 中心附近直线 draw 已明显改善；
- `w0≈±3.14` 的大旋转 curl 未覆盖；
- `h0≈±0.7` 的 guard 多数未覆盖；
- `v0=4/5/6` 的高速 takeout 未覆盖；
- 碰撞仍是简化模型。

### 9.4 出界样本

服务器将未投掷或出界壶表示为 `(0,0)`。出界样本不适合回归最终落点，但可用于训练“是否出界”的分类器或有效区域边界模型。

## 10. 擦冰策略

### 10.1 当前已上线实现

当前代码仍使用启发式/搜索擦冰：

- 部分动作明确禁止擦冰；
- 搜索模式把 sweep 当作候选维度；
- 非搜索模式根据预测短球距离估算 sweep；
- 本地物理当前仍用 `y -= 0.045 * sweep`。

禁止擦冰的主要战术包括 guard、occupy、takeout、hit-roll、clear、defense 等。

### 10.2 最新 sweep 数据

已经取得 200 条官方 sweep 样本，距离覆盖：

```text
0, 2, 4, 6, 8, 10, 12m
```

初步分析：

- x 方向平均变化接近 0；
- 0 到 8m 的平均 y 变化很小；
- 10m 和 12m 才出现明显平均前进；
- 10/12m 样本方差很大，存在少量超过 0.5m 甚至 1m 的额外前进；
- 这与当前固定线性 `0.045m/m` 不一致。

因此 sweep 数据**尚未合入正式运行配置**。需要先判断大方差来自真实非线性、倍速模式、消息时序还是采样配对问题，再采用阈值/分段模型。

## 11. 当前策略模式

### 11.1 先手

先手模型独立运行，搜索从先手视角评价局末得分。理论上倾向限制后手多分、争取偷分，但当前手工阶段先验默认关闭，实际行为主要由蒸馏模型、战术库合法性和 rollout 共同决定。

### 11.2 后手

后手使用独立 second 模型。最后一壶启用 hammer 搜索预算。当前模型尚未显式优化跨局“只得一分不如空局”的完整价值，局末奖励主要是当前端净得分。

### 11.3 比分与阶段

代码能计算 early/middle/late_setup/hammer 阶段，也有比分落后、领先时的策略先验；但因为 `strategy_gate_strength=0`，这些先验当前只保留为实验接口。

## 12. 服务器执行与部署

### 12.1 训练仓库

`DCCourse` 保存 notebook、论文、采样、拟合、训练、评测、实验模型和完整本地模拟器。

### 12.2 Runtime 仓库

`curling-runtime` 只保存：

- socket 协议；
- 当前机器人入口；
- 战术库和连续搜索；
- runtime-only 网络定义；
- 当前正式三份模型；
- 官方落点校准配置；
- 启动脚本和最小依赖。

训练仓库产出新模型或物理配置后，再显式同步到 runtime。

### 12.3 启动

```bash
export CONNECT_KEY="server_key"
export HOST="server_host"
export PORT="7788"
./start_robot.sh
```

`AIRobot` 内部会发送 `CONNECTKEY:<CONNECT_KEY>`。命令行只传裸 key；课程调试模式或魔王挑战的 `:0/:1..:8` 是 key 后缀。

## 13. 当前证据

### 13.1 本地策略评测

| 配置 | first avg | first win | second avg | second win | mean avg |
| --- | ---: | ---: | ---: | ---: | ---: |
| 共享 first 模型 | 2.9375 | 93.75% | 3.5625 | 95.00% | 3.2500 |
| first/second 独立模型 | 2.9375 | 93.75% | 3.7375 | 97.50% | 3.3375 |

对称 head-to-head：

| Policy A | Policy B | A 平均分 | A 胜率 |
| --- | --- | ---: | ---: |
| dual_refined | shared_refined | 0.25 | 50.00% |
| dual_refined | scripted | 7.05 | 97.50% |

这些结果来自旧本地简化物理，主要用于回归测试。

### 13.2 官方服务器首次实战

首次 `:1` 魔王挑战中，机器人作为 Player1 首局先手，以安全低预算 `candidates=16,rollouts=1` 被打出 `-6`。该结果暴露出：

- 本地物理与官方物理偏差大；
- rollout=1 判断噪声大；
- 大旋转、擦冰和碰撞未经官方校准；
- 本地高胜率不能直接外推到官方对手。

## 14. 已知局限

1. 正式策略模型在旧 mock 分布上训练，尚未用官方校准物理完整重训。
2. 当前官方拟合只覆盖低速、无碰撞、无擦冰、小偏移、小角速度。
3. 碰撞是最近邻简化位移，不是刚体动力学。
4. Rollout 对手偏弱，比赛搜索没有完整对手模型。
5. Value head 当前没有直接用于搜索叶节点评价。
6. 局末净得分没有完整表达多局赛制、空局和 hammer 转移价值。
7. Sweep 仍使用旧线性近似，最新数据尚未正式建模。
8. 大旋转 curl、guard 和高速 takeout 多数在校准范围之外。
9. 本地搜索时间成本较高，官方服务器时间预算尚未系统测定。
10. 官方样本仍少，缺少跨服务器启动和不同摩擦随机种子的重复验证。

## 15. 下一步优先级

### P0：完成官方物理闭环

1. 复核 sweep 样本时序和倍速设置；
2. 采集同一 `(v,h,w)` 下多次不同 sweep 的配对样本；
3. 采集 `w≈±3.14` 的 curl 样本；
4. 采集 `h≈±0.7..1.5` 的 guard/边路样本；
5. 采集 `v=4..6` 的高速路径和出界边界；
6. 单独采集正碰、偏碰和单碰撞数据。

### P1：基于官方校准环境重训

1. first/second 继续独立；
2. 使用 `balanced-mix`，保留原随机分布并引入强对手；
3. 训练新候选，不直接覆盖正式模型；
4. 用双方交换身份的 head-to-head 做晋级判断；
5. 只有在官方或高保真代理上稳定提升才替换 runtime 模型。

### P2：增强搜索

1. 用 Value head 做叶节点估值；
2. 用对手模型替换随机 rollout 对手；
3. 对连续参数使用渐进扩展；
4. 根据官方耗时动态调整候选数和 rollout；
5. 为出界风险加入显式惩罚或分类概率。

## 16. 当前推荐使用方式

首次服务器验证：

```bash
python search_distill_robot.py \
  --key "$CONNECT_KEY" \
  -H "$HOST" \
  -p "$PORT" \
  --shot-search local \
  --fixed-search \
  --search-candidates 16 \
  --search-rollouts 1 \
  --show-msg
```

该命令用于验证连接、身份、模型加载和耗时。若物理和时间稳定，再测试默认自适应预算。

每次正式实验至少保存：

```text
完整 socket 日志
每次 BESTSHOT
每次 MOTIONINFO/SWEEP
每次 POSITION
局末 SCORE
运行参数和模型 commit
```

没有这些证据时，不应根据单场比分判断某次模型或参数是否更强。
