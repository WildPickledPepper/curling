# 05 精读：KR-UCT: Monte Carlo Tree Search in Continuous Action Spaces with Execution Uncertainty

## 论文定位

这是对我们最关键的论文之一。它直接研究数字冰壶中的连续动作、执行噪声和 MCTS。论文指出，传统做法会用领域知识生成少量候选 shot，然后只在这些离散候选上搜索；问题是好 shot 可能不在候选列表中，而且某个候选的随机执行结果其实能告诉我们周围连续动作的质量。

KR-UCT 的核心贡献是：用执行噪声模型作为 kernel，在连续动作空间中共享相邻动作的价值信息，并能探索初始候选之外的新动作。

## 问题细读

冰壶动作具有：

- 连续动作：角度、速度、旋转等。
- 连续状态：石头坐标。
- 执行不确定性：投掷误差、冰面变化、石头差异。
- 对抗性：对手会响应。

传统方法：

1. 用手写 shot generator 产生候选动作，例如 draw、takeout、guard。
2. 对每个候选采样执行误差。
3. 用 rollout 或启发式评估结果。

限制：

- 只能选候选动作，不能选候选附近更好的动作。
- 某个动作的噪声样本没有用于估计其他动作。
- 候选越多，单个候选评估越浅；候选越少，又可能漏掉好动作。

## 方法细读

KR-UCT 使用 kernel regression 估计连续动作价值。对动作 `a`，由已评估动作 `b` 的统计估计：

```text
E(v | a) = sum_b K(a, b) * v_b * n_b / sum_b K(a, b) * n_b
W(a)     = sum_b K(a, b) * n_b
```

其中：

- `K(a, b)` 是 kernel。
- 论文用执行不确定性模型作为 kernel，即“如果意图动作为 a，实际执行成 b 的概率密度”。
- `E(v | a)` 代替 UCT 中的平均价值。
- `W(a)` 代替访问次数。

选择动作时使用类似 UCB 的形式：

```text
E(v | a) + C * sqrt(log(sum_b W(b)) / W(a))
```

这样做的效果：

- 相近动作共享价值信息。
- 没有被充分覆盖的区域会有更高探索奖励。
- 搜索可以在候选动作之外新增动作。

Expansion 时，KR-UCT 会在当前动作的执行噪声分布附近采样若干 outcome，选择 kernel density 较低但仍相关的位置作为新动作。这相当于沿着有希望且未充分探索的区域扩展。

## 实验细读

实验使用高保真冰壶模拟器。所有算法共享同一个领域知识 shot generator，候选通常为 7 到 30 个 shot。比较对象包括：

- 普通 UCT：只能在生成候选上搜索。
- RAVE：用动作描述共享信息。
- Progressive Widening：逐步增加动作。
- RAVE + PW。
- KR-UCT。

关键实验设置：

- 每个算法使用 1600 samples。
- rollout 最多模拟 5 壶，然后按当前局面计分。
- UCB 常数从若干值中调参，最佳多为 `C = 1.0`。
- KR-UCT 使用阈值和每次采样数控制扩展。

结果：

- KR-UCT 在一 end 对战中对所有基线有正分差，并达到统计显著。
- 对 UCT 的提升约为 0.110 分/局 end，这在冰壶中很可观。
- 最后一壶 hammer shot 分析中，KR-UCT 比其他方法多约 0.12 分以上。
- 论文图示说明：普通 UCT 被初始候选限制，KR-UCT 能从候选附近采到好区域并逐步移动到高价值区域中心。

## 擦冰细读

论文没有把擦冰作为显式动作维度。它把擦冰主要建模为降低执行误差，即让实际轨迹更接近计划轨迹。论文脚注也承认真实擦冰还有其他效果，例如让石头多走、减少 curl、末段改变落点等。

这点对我们非常重要：如果比赛协议允许擦冰，我们不能满足于“擦冰降低噪声”的简化。我们当前已把 sweep distance 加入本地 4D action，这是比论文更贴近比赛接口的一步，但物理映射还很粗。

## 对数字冰壶项目的直接启发

1. 当前 `continuous_shot_search.py` 方向正确。

   它从 tactic 候选出发，做局部连续扰动，已经是 KR-UCT 思想的轻量版。下一步可以把“独立评估候选”升级为“相邻候选共享估值”。

2. 不要只相信 tactic library。

   论文明确显示候选生成器会漏掉高价值区域。我们的 tactic library 只能做初始提案，最终动作应允许连续偏移。

3. 搜索目标要鲁棒。

   论文强调 execution uncertainty。我们本地搜索应多 rollouts，选择期望分高且方差不过大的动作。最后一壶尤其不能选窄缝高风险 shot，除非局势需要。

4. 需要官方校准。

   KR-UCT 的 kernel 来自执行模型。如果 kernel 错了，信息共享就会误导搜索。官方服务器恢复后，必须用固定 shot 采样估计速度、角度、擦冰的误差分布。

## 可实现版本

短期轻量 KR-UCT root search：

- 从模型/tactic library 取 top-k 候选。
- 在每个候选周围采样 `(v, h, w, sweep)`。
- 用高斯 kernel 根据距离共享 reward。
- 用 UCB 选择下一批采样点。
- 最终用 lower confidence bound 或均值减惩罚选 shot。

中期完整化：

- 搜索树深度至少覆盖当前壶和对手下一壶。
- 新节点用 policy model 初始化候选动作。
- rollout 使用当前模型或脚本，不用纯随机。

## 一句话结论

KR-UCT 是我们冰壶 AI 的核心论文：战术库给初始点，连续搜索负责找真正好 shot，执行噪声模型负责把局部样本泛化到邻域。我们的下一步应该是把现有局部搜索朝这个方向升级，而不是再盲目跑 DQN/PPO。
